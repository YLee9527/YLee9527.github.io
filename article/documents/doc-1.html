# 自定义路由

<Link>组件的两个主要属性：

  • href: pages目录中的路径 + 查询字符产(query string)
  • as: 展示在浏览器地址栏上的路径

例如：

  • 假设有一个 URL 为： /post/:slug

  • 先创建一个 pages/post.js 文件：

    class Post extends React.Component {
  static async getInitialProps({ query }) {
    console.log('SLUG', query.slug)
    return {}
  }
  render() {
    return <h1>My blog post</h1>
  }
}

export default Post

    server.get('/post/:slug', (req, res) => {
  return app.render(req, res, '/post', { slug: req.params.slug })

  • 使用 next/link 作为客户端路由： <Link href="/post?slug=something" as="/post/something"> 注意： 为了最佳性能会使用 <Link prefetch> , 会在连接页面的同时向后台预取数据

客户端路由的行为和浏览器完全一样：

  1. 获取组件
  2. 如果定义了 getInitialProps ,数据将被获取；如果报错，_error.js 将被渲染出来
  3. 在 1 和 2 之后，执行pushState并渲染新组件

要在你的组件中注入pathname,query或asPath，可以使用 withRouter.

# Link 强制暴露 href 给子级

如果Link子级是一个 <a> 标签并且没有 href 属性，我们将会指定他这个属性，不需要用户手动添加；但是如果 <a> 是被包装过的，那么 Link 将无法识别并传递 href给子级，需要定义一个 boolean 属性 passHref 给Link,让他强制暴露href给子级

注意：Link 中使用<a>以外的标签，并且没有通过 passHref 传递 href属性，会导致虽然可以导航正确的连接，但是当搜索引擎爬取时，将不能识别为一个链接（缺少 href 属性），可能对 SEO 优化有负面影响。

import Link from 'next/link'
import Unexpected_A from 'third-library'

function NavLink({ href, name }) {
  return (
    <Link href={href} passHref>
      <Unexpected_A>{name}</Unexpected_A>
    </Link>
  )
}

export default NavLink

# 拦截 popstate (Intercepting popstate)

有时候，需要监听 popstate 并且在路由执行前作出反应；例如，可以使用它来操作请求或者强制 SSR 刷新

import Router from 'next/router'

Router.beforePopState(({ url, as, options }) => {
  // I only want to allow these two routes!
  if (as !== '/' || as !== '/other') {
    // Have SSR render bad routes as a 404.
    window.location.href = as
    return false
  }

  return true
})

如果你传递给 beforePopState 的函数返回 false, Router 将不会处理 popstate; 你需要负责处理它。

以上 Router 对象提供以下 API：

  • route - String, 当前路由
  • pathname - String, 不包括查询参数的当前路径
  • query - Object, 解析成参数字符串，默认为{}
  • asPath - String, 展示在浏览器上的实际路径（包含参数）
  • push(url, as=url) - 使用给定的 url 执行 pushState
  • replace(url, as=url) - 使用给定的 url 执行 replaceState
  • beforePopState(cb=function) - 在 router 处理事件之前拦截 popstate

push和replace的第二个参数 as 是一个可选参数，如果在服务器上配置了自定义路由，它将很有用。

# 路由器事件(Router Events)

我们也能够监听发生在 Router 内部的不同的事件，以下为所支持事件清单：

  • routeChangeStart(url) - 当路由开始改变时触发
  • routeChangeComplete(url) - 当路由改变完成时触发
  • routeChangeError(err, url) - 当改变路由出错时触发
  • beforeHistoryChange(url) - 在更改浏览器历史记录之前触发
  • hashChangeStart(url) - 当 hash 开始改变但是页面不变时触发
  • hashChangeComplete(url) - 当 hash 改变完成但是页面不变时触发

此处的url是显示在浏览器地址栏的 url，如果你使用了 Router.push(url, as) 之类的， url 的值会是 as 的值

如何正确的监听路由事件 routeChangeStart:

const handleRouteChange = url => {
  console.log('App is changing to: ', url)
}

Router.events.on('routeChangeStart', handleRouteChange)

  if (err.cancelled) {

# 浅路由？(Shallow Routing)

Shallow routing 允许你改变 URL 的同时不会运行 getInitialProps, 你可以接收到 pathname和query的更新通过 router 属性(使用 withRouter 注入)，且不会丢失状态

也可以在调用 Router.push 和 Router.replace时使用，使用 shallow: true:

// Current URL is "/"
const href = '/?counter=10'
const as = href
Router.push(href, as, { shallow: true })
1
2
3
4

现在，当 url 更新为 /?counter=10 , Component 中的 this.props.router.query 也更新了（确保你在组件中使用了 withRouter 注入 router 属性）

可以通过 componentDidUpdate 钩子观察到改变：

componentDidUpdate(prevProps) {
  const { pathname, query } = this.props.router
  // verify props have changed to avoid an infinite loop
  if (query.id !== prevProps.router.query.id) {
    // fetch data based on the new query
  }
}

# 使用高阶组件(Using a Higher Order Component)

如果想在应用中任何一个组件内部访问 router 对象，需要使用 withRouter 高阶组件：

import { withRouter } from 'next/router'

function ActiveLink({ children, router, href }) {
  const style = {
    marginRight: 10,
    color: router.pathname === href ? 'red' : 'black'
  }

  const handleClick = e => {
    e.preventDefault()
    router.push(href)
  }

  return (
    <a href={href} onClick={handleClick} style={style}>
      {children}
    </a>
  )
}

export default withRouter(ActiveLink)

  return (
    <div>

}

以下例子，使 /a 解析 /pages/b, 使 /b 解析 /pages/a

// This file doesn't go through babel or webpack transformation.
// Make sure the syntax and sources this file requires are compatible with the current node version you are running
// See https://github.com/zeit/next.js/issues/1245 for discussions on Universal Webpack or universal Babel
const { createServer } = require('http')
const { parse } = require('url')
const next = require('next')

const dev = process.env.NODE_ENV !== 'production'
const app = next({ dev })
const handle = app.getRequestHandler()

app.prepare().then(() => {
  createServer((req, res) => {
    // Be sure to pass `true` as the second argument to `url.parse`.
    // This tells it to parse the query portion of the URL.
    const parsedUrl = parse(req.url, true)
    const { pathname, query } = parsedUrl

    if (pathname === '/a') {
      app.render(req, res, '/b', query)
    } else if (pathname === '/b') {
      app.render(req, res, '/a', query)
    } else {
      handle(req, res, parsedUrl)
    }
  }).listen(3000, err => {
    if (err) throw err
    console.log('> Ready on http://localhost:3000')
  })
})

# 禁用文件系统路由(Disabling file-system routing)

# 动态 assetPrefix(Dynamic assetPrefix)

有时候需要动态的设置 assetPrefix。这是很有用的基于传入的请求来改变 assetPrefix，为此，可以使用 app.setAssetPrefix。用法示例：

const next = require('next')
const http = require('http')

const dev = process.env.NODE_ENV !== 'production'
const app = next({ dev })
const handleNextRequests = app.getRequestHandler()

app.prepare().then(() => {
  const server = new http.Server((req, res) => {
    // Add assetPrefix support based on the hostname
    if (req.headers.host === 'my-app.com') {
      app.setAssetPrefix('http://cdn.com/myapp')
    } else {
      app.setAssetPrefix('')
    }

    handleNextRequests(req, res)
  })

  server.listen(port, err => {
    if (err) {
      throw err
    }

    console.log(`> Ready on http://localhost:${port}`)
  })
})

import React from 'react'

    return { pageProps }
  }

  render() {
    const { Component, pageProps } = this.props

    return (
      <Container>
        <Component {...pageProps} />
      </Container>
    )
  }
}

# 自定义错误处理(Custom error handling)

404 和 500 通过默认组件error.js在客户端和服务器端同时处理，若要覆盖它需要在 pages 目录下定义一个 _error.js。

⚠️ pages/_error.js 仅在生产环境生效。在开发环境中，您会得到一个带有调用堆栈的错误，以了解错误来自何处。

import React from 'react'

class Error extends React.Component {
  static getInitialProps({ res, err }) {
    const statusCode = res ? res.statusCode : err ? err.statusCode : null
    return { statusCode }
  }

  render() {
    return (
      <p>
        {this.props.statusCode
          ? `An error ${this.props.statusCode} occurred on server`
          : 'An error occurred on client'}
      </p>
    )
  }
}

export default Error

# 使用备用主机名启动服务器(Starting the server on alternative hostname)

使用不同的默认主机名启动开发服务器，可以使用 --hostname hostname_here 或者 -H hostname_here 的 next 开发选项。这将启动一个 TCP 服务器，监听所提供主机上的连接。

# CDN 支持资源前缀

设置 CDN，可以设置 assetPrefix 并且配置 CDN 的源

const isProd = process.env.NODE_ENV === 'production'
module.exports = {
  // You may only need to add assetPrefix in the production.
  assetPrefix: isProd ? 'https://cdn.mydomain.com' : ''
}

注意:Next.js 将自动在它加载的脚本中使用这个前缀，但这对/static 没有任何影响。如果希望通过 CDN 提供这些资源，就必须自己引入前缀。在本例中记录了一种引入前缀的方法，该前缀在组件中工作，并随环境而变化。

如果你的 CDN 在一个单独的域中，并且想通过 CORS 感知请求资源，需要设置如下配置项：

// next.config.js
module.exports = {
  crossOrigin: 'anonymous'
}

# 生产部署(Production deployment)

要部署，而不是运行 next 需要提前为生产构建。因此，构建和启动是分开的的命令：

next build
next start
